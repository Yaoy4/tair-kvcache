## 2. 通话核心摘要

### 2.1 项目背景

Jessie 介绍，Intel GPU accelerator 组主要服务于 data center AI inference 场景。团队里有一部分人关注 GPU / silicon architecture，目标是指导下一代 GPU 如何设计，才能更好地服务 AI inference workload。

当前 James 侧重点在 GPU architecture 和 performance modeling 上，使用 AI Configurator 较多。AI Configurator 可以基于已有 performance database 预测配置和性能，不一定需要真实 GPU 运行。

但是 Jessie 强调，Intel 需要看的不只是单个 GPU 的 kernel performance，而是从 customer workload 的真实运行环境出发，理解整个 AI workload system 的性能瓶颈。

---

### 2.2 为什么要做 system-level simulation

客户真实环境中的瓶颈可能不在单个 GPU 上，而可能来自：

- CPU 编排、调度和 offload；
- GPU kernel performance；
- GPU-GPU / node-node communication；
- PCIe、Ethernet、NVLink 等 interconnection；
- 多 GPU、多节点拓扑；
- MoE expert offload 到 CPU 或其他设备；
- Intel GPU 与 NVIDIA GPU 等异构硬件混合部署；
- inference engine / serving framework 差异。

因此，Intel 希望构建一个系统级仿真环境，在没有完整真实硬件环境的情况下，快速回答：

> 一个客户 workload 在某种 CPU + GPU + interconnect + offload 组合下运行，系统瓶颈在哪里？  
> 如何调整系统配置，才能获得更好的 end-to-end performance？  
> 对未来 Intel GPU / CPU / software stack 的设计有什么建议？

---

## 3. 技术需求拆解

### 3.1 当前已有基础

从通话内容看，现有基础大概包括：

- HiSim：已有部分 GPU / serving / KV cache offloading 相关仿真能力；
- AI Configurator：已有基于 performance database 的配置和性能预测能力；
- James 可能已有若干 Intel GPU 或 NVIDIA GPU 的 performance database；
- 现有能力更偏向单 GPU 或单设备视角；
- system-level view 仍然不足。

---

### 3.2 需要补齐的关键能力

#### 能力 1：Interconnection 仿真

你在通话中主动提出，可以优先考虑 CPU-GPU、GPU-GPU、GPU-node 之间的通信仿真。这个判断是对的，而且应该作为 POC 的第一优先级。

需要考虑的通信类型包括：

- CPU ↔ GPU；
- GPU ↔ GPU；
- Node ↔ Node；
- PCIe；
- Ethernet；
- NVLink；
- 远端 DDR / HBM / KV cache offload 路径。

需要输出的指标包括：

- bandwidth；
- latency；
- transfer time；
- communication overhead；
- communication/computation overlap；
- 对 end-to-end latency 的影响。

---

#### 能力 2：异构硬件性能数据库整合

Jessie 明确提到，未来客户环境可能不是纯 Intel GPU，也可能是 Intel GPU + NVIDIA GPU 混合，或者 Intel CPU + 第三方 accelerator 混合。

因此需要统一描述不同硬件的性能数据：

- Intel GPU；
- NVIDIA GPU，如 H20 等；
- Intel CPU；
- 不同 memory hierarchy；
- 不同 interconnect；
- 不同 inference workload 下的 performance profile。

这一步的重点不是一开始收集很多真实数据，而是先设计一个能扩展的 schema，让 demo 可以跑起来。

---

#### 能力 3：Workload 建模

Jessie 希望 simulation 面向真实 workload，而不是只看单个 kernel。

可以先从简单 workload 开始：

- 单模型 inference；
- batch size 不同；
- prompt length / output length 不同；
- MoE expert offload；
- KV cache offload；
- 多 GPU tensor / pipeline / data parallel 的简单抽象；
- CPU preprocessing / scheduling / orchestration。

第一版 POC 不需要完整模拟复杂 serving stack，但需要有清晰的 workload DAG 或 execution graph。

---

#### 能力 4：System-level 瓶颈分析

最终 demo 不能只是给出运行时间，而是要能解释：

- 总 latency 分成哪些部分；
- compute 占多少；
- communication 占多少；
- CPU overhead 占多少；
- memory / offload overhead 占多少；
- 哪个 component 是 bottleneck；
- 换一种配置后，瓶颈是否转移；
- 对硬件和软件设计有什么启发。

这也是这个项目最有价值的部分。

# Objective 2：入职后 2 周内完成一个最小可行 POC

## KR1：定义一个最小 workload model

### 衡量标准

第一版 workload 不追求真实复杂，只需要能表达：

- compute stage；
- communication stage；
- CPU stage；
- GPU stage；
- memory/offload stage；
- stage dependency。

### 你应该做什么

先不要上来就模拟完整大模型 serving。可以定义一个简化 workload graph：

```text
Prompt preprocessing on CPU
        |
        v
Prefill compute on GPU
        |
        v
GPU-GPU communication
        |
        v
Decode compute on GPU
        |
        v
KV cache transfer / offload
        |
        v
Output postprocessing on CPU
```

每个 stage 只需要有：

- stage name；
- device；
- input size；
- compute time；
- communication time；
- dependency；
- output metric。

---

## KR2：实现一个可替换的 interconnect model

### 衡量标准

能通过配置切换不同通信链路：

- PCIe；
- NVLink；
- Ethernet；
- custom bandwidth / latency。

### 你应该做什么

先实现简单公式：

```text
communication_time = fixed_latency + data_size / bandwidth
```

后续再扩展：

- contention；
- topology；
- overlap；
- collective communication；
- all-reduce；
- all-to-all；
- MoE routing communication。

第一版先做简单模型即可，重点是 architecture 能扩展。

---

## KR3：把 compute performance 与 performance DB 解耦

### 衡量标准

不同 device 的 compute time 不要写死在代码里，而是从配置文件或 database 读取。

建议 schema：

```yaml
devices:
  intel_gpu_a:
    type: gpu
    vendor: intel
    compute_profile:
      prefill_time_ms: 12.0
      decode_time_ms: 2.5
  nvidia_h20:
    type: gpu
    vendor: nvidia
    compute_profile:
      prefill_time_ms: 9.0
      decode_time_ms: 2.0

interconnects:
  pcie:
    bandwidth_GBps: 32
    latency_us: 2
  nvlink:
    bandwidth_GBps: 100
    latency_us: 1
  ethernet:
    bandwidth_GBps: 12.5
    latency_us: 10
```

---

## KR4：输出一份 bottleneck report

### 衡量标准

POC 跑完后，能够输出类似下面的结果：

```text
Total latency: 38.5 ms

Breakdown:
- CPU preprocessing: 2.0 ms
- GPU compute: 21.0 ms
- GPU-GPU communication: 10.5 ms
- KV cache offload: 3.0 ms
- CPU postprocessing: 2.0 ms

Bottleneck:
- GPU compute is the largest component.
- Communication is the second largest component.
- If using NVLink instead of PCIe, total latency decreases by 18%.
```

这类输出比单纯给一个 latency 数字更有价值。

你在通话中提到“不同 prompt 输入”作为 workload 差异，这个方向可以保留，但要表达得更专业。

- decode length；
- batch size；
- model size；
- MoE expert routing；
- KV cache size；

这样会比“不同输入 prompt”更贴合 system simulation。
