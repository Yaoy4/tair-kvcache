The demo tool is hisim/tools/pd_demo.py. It runs the same Backend A pipeline used in production but standalone (no SGLang server needed), so we can sweep knobs in seconds.

How to read the output

Every run prints three blocks. Here's a sample header:

prefill role     : device=h100_sxm  tp=1  replicas=2
decode  role     : device=h100_sxm  tp=1  replicas=2
KV transfer      : bw=100.0 GB/s, latency=10.0 us
kv_bytes_per_tok : 131072  (derived via utils.calc_kv_cache_cell_elems)

Then per-request rows:

rid=r4  in=2048 out=48  TTFT=4.05 ms  KV=2.69 ms  E2E=4.71 ms
                        pq=0.51  kvt=2.69  dq=0.00 ms

Columns: - TTFT = time from arrival to first output token (= prefill_queue + prefill_compute + kv_transfer + first decode step). - KV = the KV-transit time (bandwidth + latency). - E2E = arrival → finished, the full request. - pq (prefill_queue_wait) = how long the request sat before any prefill replica was free. - kvt (kv_transfer_time) = same as KV; broken out so you can aggregate it. - dq (decode_queue_wait) = how long the request sat after KV-ready before a decode batch slot opened up.

And a footer:

Phase 3 stage percentiles (via calc_metrics):
  prefill_queue ms : mean=0.41  p50=0.36  p95=1.06  p99=1.24
  kv_transfer   ms : mean=1.18  p50=0.93  p95=2.46  p99=2.65
  decode_queue  ms : mean=0.00  p50=0.00  p95=0.00  p99=0.00

These percentiles are the exact same metric the production sglang_hook surfaces in real HiSim runs. The demo is a faithful in-process preview.

Step 1 — Baseline (balanced disagg)

python hisim/tools/pd_demo.py --requests 8 --burst \
  --prefill-replicas 2 --decode-replicas 2 --bw-gbps 100

Result (averages):

TTFT KV-transit E2E prefill_queue p95 kv_transfer p95 decode_queue p95
1.97 ms 1.18 ms 3.02 ms 1.06 ms 2.46 ms 0.00 ms

Reading it: balanced 2/2 with fast KV. Nothing is queueing on the decode side. The longest request (in=2048) absorbs ~0.5 ms of prefill queue + 2.7 ms of KV transfer — KV transfer is already the dominant TTFT component, not the prefill compute.

Step 2 — Slow KV link (10 GB/s instead of 100)

python hisim/tools/pd_demo.py --requests 8 --burst \
  --prefill-replicas 2 --decode-replicas 2 --bw-gbps 10
TTFT KV-transit E2E kv_transfer p95
12.53 ms 11.75 ms 13.55 ms 24.50 ms

Insight: halving bandwidth ~doubles kv_transfer_p95; 10× slower bandwidth → 10× slower transfer. KV transfer time goes from 39% of TTFT (step 1) to 94% of TTFT here. In disagg, the inter-node KV link is a first-class TTFT driver — far more impactful than the prefill compute itself for typical context lengths.

Decision rule: if your interconnect cannot match input_length × kv_bytes_per_token / target_TTFT_budget, disagg loses to agg every time. With 131 kB/token at 1024-token prompts, you need >10 GB/s just to keep KV under 13 ms.

Step 3 — Decode-bottleneck (4 prefill replicas, 1 decode)

python hisim/tools/pd_demo.py --requests 8 --burst \
  --prefill-replicas 4 --decode-replicas 1 --bw-gbps 100
TTFT E2E decode batching prefill_queue p95
1.68 ms 2.91 ms 261 batch-steps, avg batch size 2.45 (max 5) 0.34 ms

Insight: TTFT actually improved (1.97 → 1.68 ms) because we have more prefill replicas. But the real story is decode batching went from 1.19 → 2.45 average batch size. With only one decode replica, requests pile up and get batched together — which is exactly what TPOT cares about. TPOT improves with batching up to the GPU's compute roofline, then degrades. This is why production disagg tunes decode replicas to land in the optimal batch-size sweet spot.

decode_queue_p95 stays at 0 because with a tiny 8-request workload, batching absorbs all the waiting. Crank --requests 64 and you'd see it grow.

Step 4 — Prefill-bottleneck (1 prefill, 4 decode) — the opposite

python hisim/tools/pd_demo.py --requests 8 --burst \
  --prefill-replicas 1 --decode-replicas 4 --bw-gbps 100
TTFT E2E decode batching prefill_queue p95
2.63 ms 3.64 ms 640 batch-steps, avg batch size 1.00 2.40 ms

Insight: mirror image. prefill_queue_p95 jumped 7× (0.34 → 2.40 ms) — requests waiting for the single prefill replica is now dominant. Decode batches are size-1 (over-provisioned), so TPOT is best-case but you're wasting decode silicon. TTFT degrades by 57% even though decode is faster — because all 8 requests serialize through one prefill worker.

The lesson Step 3 vs Step 4: prefill replicas drive TTFT, decode replicas drive TPOT batching efficiency. Disagg lets you tune them independently; agg cannot.

Step 5 — Heterogeneous (H100 prefill, H20 decode)

python hisim/tools/pd_demo.py --requests 8 --burst \
  --prefill-device h100_sxm --decode-device h20 \
  --prefill-replicas 2 --decode-replicas 2 --bw-gbps 100
TTFT E2E
1.99 ms 4.21 ms

Insight: TTFT is identical to Step 1's baseline (prefill is on the same H100s), but E2E jumped 40% (3.02 → 4.21 ms) because decode is now on slower H20s. This is the cost-efficiency play real disagg deployments make: use cheap, lower-bandwidth chips for the memory-bound decode phase (where you'd be bandwidth-starved on premium chips anyway), and reserve premium compute for prefill. The demo lets you quantify the TTFT/TPOT trade before buying hardware.

Step 6 — Compare disagg vs agg (the "do I even need this?" question)

For an apples-to-apples agg comparison: set identical roles, huge bandwidth (eliminates KV transit), and one combined pool. The Phase 4 smoke test hisim/tests/unit/simulation/test_pd_backend_a_smoke.py verifies this analytically — with zero-KV identical roles, Backend A E2E equals prefill_dur + N × decode_dur (the serial baseline).

In practice: disagg wins when

  1. KV bandwidth is sufficient (kv_transfer << TTFT budget),
  2. Decode benefits from large batches (batch size > 4-8 typical),
  3. You can run prefill and decode on different SKUs.

Agg wins when

  1. Interconnect is slow (KV transit dominates),
  2. Workload is short-output (TPOT doesn't matter, batching gives no win),
  3. Hardware is homogeneous and small-scale (orchestration overhead > parallelism win).

Cheat sheet: which knob impacts which metric

Knob Affects primarily Why
--prefill-replicas prefill_queue_*, TTFT Fewer replicas → requests serialize on prefill
--decode-replicas decode_queue_*, TPOT batching More replicas → smaller batches → worse TPOT-per-cost
--bw-gbps kv_transfer_*, TTFT, E2E KV transit is input_length × bytes / bw
--latency-us kv_transfer_* (small reqs only) Fixed overhead per transfer
--prefill-device TTFT (via prefill_compute) Device speed in the analytic predictor / AIC DB
--decode-device TPOT, E2E Each decode step latency
--prefill-tp, --decode-tp prefill_compute, decode_compute TP discount in the predictor
--requests, --burst All queueing metrics More concurrent load → real queueing forms

Going from demo to real HiSim

The demo uses an analytic stub predictor. For real AIConfigurator-backed numbers:

# 1) AIConfigurator gives you a top-N CSV. Pick a row and convert:
python hisim/tools/aic_to_hisim_bridge.py \
  --aic-csv pareto.csv --row-index 0 --output hisim_disagg.json \
  --database-path /aic/db \
  --emit-disagg --kv-transfer-bw-gbps 50 --kv-transfer-latency-us 20

# 2) Quick standalone sanity check (loads the real config):
python hisim/tools/pd_demo.py --config hisim_disagg.json --requests 8 --burst

# 3) Full HiSim through SGLang hook with real workload:
bash hisim/run_hisim_demo.sh  # uses the same JSON path

The full HiSim path emits the same prefill_queue_ms / kv_transfer_ms / decode_queue_ms percentile fields in metrics.json that you see in the demo footer — so the demo insights translate directly to production interpretation.

TL;DR insights to take to a review meeting

  1. KV bandwidth is the make-or-break knob. It often dominates TTFT in disagg. Measure your real interconnect before claiming disagg wins.
  2. Prefill replicas → TTFT; decode replicas → TPOT batching efficiency. This decoupling is the entire reason disagg exists.
  3. Heterogeneous deployments are quantifiable now. The H100/H20 split is a real cost play; HiSim tells you the latency cost before procurement.
  4. Stage percentiles isolate root cause. When TTFT regresses, look at prefill_queue_p95 (need more prefill) vs kv_transfer_p95 (need faster link). When TPOT regresses, look at decode batch size (too many decode replicas → batches too small → wasted compute).
